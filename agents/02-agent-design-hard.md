# 译文: Agent 设计依然困难重重

## 前言

本文侧重于 Agent 的工程实践，覆盖了缓存设计、上下文管理、工具调用、子 Agent 协作等多个关键问题。内容并非理论推导，而是基于真实系统构建过程中的经验总结，对于正在设计或优化 Agent 系统的开发者具有很强的参考意义。

阅读指南:

- 如果你正在构建或优化 Agent 系统，建议重点关注缓存（Caching）、上下文管理、强化学习以及工具调用相关章节。
- 如果你更关心架构设计，可以重点阅读子 Agent、文件系统共享状态以及输出工具设计部分。
- 若你关注工程落地与稳定性问题，测试与评估、故障隔离等内容会更有参考价值。

本文内容偏实践导向，适合有一定经验的开发者阅读，新手也可以结合实际项目逐步理解。

## 正文

我觉得现在是时候总结一下我最近学到的新经验了。本文大部分内容将围绕 **构建 Agent** 展开，同时讨论如何使用 Agent 编程工具。

构建 Agent 的过程依旧混乱。一旦涉及真实工具调用，SDK 的抽象层往往会失效。自行管理缓存（Caching）效果更佳，但不同模型的缓存机制差异极大。强化学习承担的工作量往往超出预期，并且必须严格隔离故障，避免破坏执行流程。通过类文件系统层实现共享状态是构建 Agent 的重要模块。输出工具的设计难度出乎意料，而模型选择依然需要根据具体任务权衡。

## 该选用哪款 Agent SDK？

在构建自研 Agent 时，你可以选择直接基于底层 SDK（如 OpenAI SDK 或 Anthropic SDK）开发，也可以采用更高阶的抽象层，例如 Vercel AI SDK 或 Pydantic。我们此前的选择是使用 **Vercel AI SDK**，但仅利用其 provider 抽象层，Agent 循环基本由我们自己驱动。如今回过头看，我们不会再做这样的选择。Vercel AI SDK 本身毫无问题，但在尝试构建 Agent 时，有两件事是我们最初未曾预料到的：

1. **不同模型之间的差异显著**：你需要构建自己的 Agent 抽象层。我们尚未发现 SDK 中存在适配 Agent 的理想抽象层。我认为部分原因在于：尽管基础 Agent 设计只是一个循环，但不同工具的特性会导致细微差异。这些差异会影响理想抽象层的实现难易程度，包括缓存控制、强化学习的不同需求、工具提示词、provider 侧工具等。由于理想抽象层尚未明确，使用原生 SDK 可以获得完全控制权。而采用高阶 SDK 时，你必须在其现有抽象层之上开发，最终架构可能并非你真正想要的。

2. **LLM provider 侧工具集成挑战大**：例如，Anthropic 的网络搜索工具在搭配 Vercel SDK 使用时，经常破坏消息历史记录，目前原因尚未完全查明。此外，就 Anthropic 而言，直接对接其原生 SDK 而非 Vercel SDK，缓存管理更简单，出错时的报错信息也更加清晰。

这种情况或许会改变，但目前我们在构建 Agent 时大概率不会使用抽象层，至少在行业格局稳定之前是如此。对我们而言，其收益尚未超过成本。

当然，也可能其他人已经找到解决方案。如果你读到这里并认为我的观点有误，欢迎发邮件给我，我非常乐意学习。

> 参考 https://github.com/badlogic/pi-mono 项目为兼容各家 LLM API 做的工作。

## 缓存实践心得

不同平台的缓存策略差异巨大。已有诸多讨论，但 Anthropic 采用的付费缓存模式要求用户显式管理缓存节点，这从 Agent 工程层面彻底改变了交互方式。我最初觉得手动管理缓存很不合理：为什么平台不能代劳？但现在我完全改变了观点，更偏爱显式缓存管理。它让成本和缓存利用率更可预测。

显式缓存能够实现一些其他方式难以完成的操作，例如：  

- 拆分一段对话，让其同时沿两个方向推进  
- 进行上下文编辑  

尽管最优策略尚不明确，但显然提供了更多控制权，而我非常享受这种掌控感。同时，它也让底层 Agent 的成本更容易核算，你能更准确地预估缓存利用率，而其他平台的缓存效果则时好时坏。

在 Anthropic Agent 中，我们的缓存实现方式如下：  

- 系统提示词后设置一个缓存节点  
- 对话开头设置两个缓存节点，最后一个缓存节点会随对话尾部向上移动  
- 过程中可以做一些优化  

由于系统提示词和工具选择现在基本需要保持静态，我们会在后续传入动态消息以提供当前时间等信息，否则会破坏缓存。同时，我们在循环中更多地运用了强化学习。

> Prompt Caching 是 LLM 架构层面最大的创新之一。 它打破了“无状态 API”的魔咒，让长上下文应用变得既便宜又快速。
> ref: https://yeasy.gitbook.io/claude_guide

## Agent 循环中的强化学习

在每次 Agent 运行工具时，你不仅可以返回工具生成的数据，还能向循环中注入额外信息。例如：  

- 提醒 Agent 整体目标及各子任务状态  
- 在工具调用失败时提供可能成功的提示  
- 向系统告知后台状态变化  

有时 Agent 自我强化就足够了。以 Claude Code 为例，待办事项写入工具就是一个自我强化工具。它仅接收 Agent 认为需要执行的任务列表，然后原样输出内容。本质上只是一个回显工具，除此之外没有其他功能。但相比于仅在上下文开头给出任务和子任务，且期间发生大量操作的情况，这已足以更好地推动 Agent 执行。

强化学习还可以向系统告知执行过程中环境是否发生不利变化。例如，如果 Agent 执行失败并从某一步骤重试，但恢复操作基于损坏的数据，我们会注入一条消息，提示其可能需要回退几步，重新执行之前的步骤。

## 故障隔离

如果预期代码执行过程中会出现大量故障，可以将这些故障从上下文中屏蔽。主要有两种实现方式：

第一种是将可能需要迭代的任务单独运行，通过子 Agent 执行直至成功，仅返回成功结果，并附带一份无效方案的简要总结。Agent 了解子任务中的失败案例是有帮助的，因为这些信息可以被应用到后续任务中，从而避免重复犯错。

第二种方式并非适用于所有 Agent 或基础模型，但在 Anthropic 中可以进行上下文编辑。截至目前，我们在上下文编辑方面尚未取得太多成果，但我们认为这是一个值得深入探索的方向，也很希望了解其他人是否在这方面取得了进展。上下文编辑的优势在于，它可以为后续迭代循环节省令牌。你可以从上下文中移除那些无助于循环成功完成、仅对执行过程中的部分尝试产生负面影响的故障信息。但正如前面所提到的，让 Agent 了解失败原因同样是有价值的，因此或许没有必要保留所有故障的完整状态和输出信息。

遗憾的是，上下文编辑会自动使 Caching 失效，这一点无法避免。因此，这种取舍是否能够抵消 Caching 失效带来的额外成本，目前仍不明确。

## 子 Agent

正如我在本博客中多次提到的，我们的大多数 Agent 都基于代码执行和代码生成，这就要求 Agent 拥有一个公共的数据存储位置。我们选择了文件系统——更准确地说，是虚拟文件系统——但这需要不同的工具能够访问它。在涉及子 Agent 或子推理时，这一点尤为重要。

你应该构建一个没有“死胡同”的 Agent。所谓死胡同，是指任务只能在某个你开发的子工具内部继续执行。例如，你可能开发了一个生成图片的工具，但它只能将图片传递给下一个工具。这会带来问题，因为你之后可能希望通过代码执行工具将这些图片打包成压缩包。因此，你需要一个系统，让图片生成工具可以将图片写入一个代码执行工具能够读取的位置，本质上就是文件系统。

显然，反向操作同样需要支持。你可能希望通过代码执行工具解压压缩包，然后回到推理环节对所有图片进行描述，接着再回到代码执行，如此循环往复。文件系统正是我们实现这一流程的机制，但这要求工具在设计时支持接收虚拟文件系统中的文件路径来完成操作。

简单来说，代码执行工具和推理运行工具会访问同一个文件系统，而推理运行工具可以读取该虚拟文件系统中指定路径的文件。

## 输出工具的使用

我们 Agent 架构的一个有趣之处在于，它并不是一个传统的聊天会话。Agent 最终会向用户或外部系统传递信息，但中间过程中的所有消息通常不会对外展示。问题在于：它如何生成最终消息？

我们设计了一个输出工具，Agent 通过它显式与人类交互，并通过提示词指导其在何时使用该工具。在我们的场景中，输出工具用于发送邮件。

然而，这带来了一些新的挑战。首先，与直接使用 Agent 主循环的文本输出与用户沟通相比，控制输出工具生成内容的措辞和语气要困难得多。我无法完全解释原因，但推测这可能与模型的训练方式有关。

我们曾尝试让输出工具调用像 Gemini 2.5 Flash 这样的轻量模型来调整语气，但这不仅增加了延迟，反而还降低了输出质量。其中一个原因是主模型本身的措辞不够准确，而子工具又缺乏足够的上下文信息。向子工具注入更多主 Agent 的上下文片段会导致成本急剧上升，同时也无法彻底解决问题，甚至有时会在最终输出中泄露不必要的信息，例如达成最终结果的执行步骤。

输出工具的另一个问题是，Agent 有时不会调用该工具。我们的一个强制策略是记录输出工具是否被调用：如果循环结束时没有调用输出工具，就注入一条强化消息，引导 Agent 使用该工具。

## 模型选择

总体而言，我们的模型选择至今没有发生太大变化。我认为 Haiku 和 Sonnet 依然是最优的工具调用模型，非常适合用于 Agent 循环，其强化学习逻辑也相对透明。另一个显而易见的选择是 Gemini 模型。截至目前，GPT 系列模型在主循环中的表现并不理想。

对于部分同样需要推理的子工具，如果需要总结大型文档或处理 PDF 等文件，我们目前选用 Gemini 2.5。该模型在图片信息提取方面也表现出色。相比之下，Sonnet 系列模型经常触发安全过滤，这在实际使用中带来了较大影响。

还有一个同样明显的结论：仅凭 Token 成本无法真正衡量 Agent 的整体开销。更优秀的工具调用模型可以用更少的 Token 完成任务。如今市面上确实存在比 Sonnet 更便宜的模型，但在循环执行场景中，它们未必更具成本优势。

但总体来看，近几周行业格局并未发生明显变化。

## 测试与评估

我们发现，测试与评估是当前最大的难题。这并不令人意外，但 Agent 的特性使这一问题更加棘手。与提示词不同，你无法在外部系统中完成评估，因为需要注入的信息过多。这意味着评估必须基于可观测性数据或实际运行过程中的埋点。

截至目前，我们尝试过的所有方案都未能让我们认可其可行性。遗憾的是，我们仍然没有找到一个令人满意的解决方案。希望未来能够有所突破，因为这已经成为构建 Agent 过程中越来越令人困扰的一环。

## 编程 Agent 最新进展

就我个人使用编程 Agent 的体验而言，并没有太大变化。最近的主要变化是我开始更多地尝试使用 Amp。如果你好奇原因，这并不是因为它在客观上比我当前使用的 Agent 更优秀，而是因为从其发布的内容来看，我非常认同他们在 Agent 设计上的思路。

例如，Oracle 等不同子 Agent 与主循环之间的交互设计非常精妙，目前很少有其他框架能够做到这一点。这也成为我用来验证不同 Agent 设计效果的一种有效方式。

与 Claude Code 类似，Amp 给人的感觉是一款由开发者自用驱动的产品，而行业内并非所有 Agent 都具备这样的特质。
