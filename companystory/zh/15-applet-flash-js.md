# 从安全视角看客户端技术

回看客户端技术的发展史，会发现一个比较有意思的现象：一些技术在某个阶段几乎不可替代，却在短时间内迅速退出舞台；而另一些技术，即便问题不断，
却始终能够演化并延续下来。

用安全视角理解这个过程，会得到一个比较好的解释: 决定技术生死的，不只是能力本身，而是它如何被约束：权限是否被清晰界定，执行环境是否具
备足够的隔离能力。

## ActiveX 

ActiveX 是 CS 架构最早的尝试。它本质上是基于 COM 的本地组件，运行时直接侵入用户空间，可以调用操作系统提供的各类能力。对开发者而言，这意味着
浏览器页面可以获得几乎“原生应用级别”的能力。

但从安全角度看，这种设计等于绕过了浏览器本应承担的隔离职责。用户一旦授权，程序就可以读写文件、调用系统接口、启动进程......

攻击者不必依赖复杂漏洞，用网页就能控制你的电脑。

## Applet

Java Applet 试图走一条不同的路径。它引入了沙箱模型，希望在“可执行代码”和“系统能力”之间加一层边界。从理念上说，这已经比 ActiveX 前进了一步。
但问题在于，这个沙箱并不可靠：历史上多次出现逃逸漏洞，一旦突破边界，结果与直接运行本地代码并无本质区别。随着漏洞积累和维护成本上升，Applet 
最终被边缘化，也就不难理解了。

## Flash

Flash 的路径又有所不同。它并不是完全无约束的执行环境，也有权限提示和一定的限制机制，例如访问摄像头或麦克风需要用户授权，本地存储也被限制在特定
范围内。但它的问题不主要在权限模型，而在于实现复杂度和历史包袱：长期演进形成的大量攻击面，使其频繁成为漏洞利用的载体。

以上技术的共同点在于，它们把浏览器当成了“加载器”，而不是“边界”。程序一旦启动，后续就只能依赖操作系统和运行时本身的安全性。

## JavaScript

JavaScript 的演化路径恰好相反。它并不是因为“天生安全”而存活下来，而是因为始终被限制在浏览器定义的执行模型之内。浏览器不仅提供 API，更重要的是
定义了能力的边界。JavaScript 可以操作 DOM、发起网络请求、使用本地存储，但这些能力都通过浏览器这一层被重新封装。

同源策略是这种约束的一部分，但它常被过度简化。更准确地说，JavaScript 可以向任意地址发起请求，但是否能够读取响应内容，取决于目标服务是否显式允许。
这种设计并不是阻止通信本身，而是阻止数据被任意读取，从而限制跨站数据泄露的可能性。

类似的约束也体现在本地资源访问上。JavaScript 不能直接遍历文件系统，也无法在没有用户参与的情况下读取本地文件；即便是通过文件选择框获得的访问权限，
也是一次性的、受控的。浏览器并不是完全拒绝能力，而是把能力拆分成更细粒度的授权单元，并且把最终决策权交给用户或策略系统。

不过，把 JavaScript 描述为“几乎无法造成伤害”同样是不准确的。它确实很难直接突破到操作系统层面，但在浏览器这一层，依然可以造成严重后果。跨站脚本
攻击可以窃取用户会话、操作账户，甚至借助同源环境执行任意业务逻辑。换句话说，它限制的是“系统级破坏”，而不是“数据与身份层面的风险”。

这种差异恰恰说明，现代浏览器安全模型的目标并不是消除风险，而是把风险限制在一个可控范围内。即使攻击发生，其影响也被尽量约束在单一站点或会话之内，而
不会轻易扩散到整个操作系统。

从这个角度看，WebAssembly 的出现并没有改变方向。它提供了接近原生性能的执行能力，但仍然运行在浏览器的沙箱中，不能直接访问系统资源，所有外部交互都需
要通过既有的 Web API 完成。性能被提升了，但权限模型并没有被放松，这其实延续了同一条设计原则。

## 总结

如果把这些技术串在一条时间线上，可以看到一个逐渐收敛的趋势：客户端计算能力在增强，但权限和隔离机制在严格限制其行为。浏览器不再只是内容展示工具，而是
设置了一个明确的安全边界，所有进入其中的执行逻辑，都必须接受这套边界的约束。

因此，与其说某些技术“被淘汰”，不如说它们所代表的那种“越过边界获取能力”的模式被淘汰了。在今天的环境下，客户端计算仍会持续演进，但只要运行在开放网络之上，
权限边界与隔离机制就不会退场。性能与安全也不再是对立关系，而是在约束之中协同推进, 我们拥有更强的能力，同时也在更严格的边界内使用这些能力。这种在约束中
释放能力的方式，本质上就是一种 Harness engineering。
