# Pi：OpenClaw 中的极简 Agent

如果你最近没有与世隔绝，大概已经注意到，本周我朋友 Peter 的一个项目在互联网上突然爆红。这个项目有过很多名字。最近的名字叫 **OpenClaw**，但如果你在新闻中看到它，也可能叫 **ClawdBot** 或 **MoltBot**——取决于你是在什么时候看到相关报道的。它本质上是一个连接到你选择的通信渠道的 agent，然后直接运行代码。

但你可能不太熟悉的是：**OpenClaw 的内部核心其实是一个名叫 Pi 的小型编码 agent**。而在目前为止，它几乎已经成为我唯一在使用的编码 agent。过去几周里，我几乎变成了这个小 agent 的“布道者”。最近我做了一场关于它的演讲之后才意识到，我其实还没有在这个博客里真正写过 Pi。所以我觉得应该写一点内容，解释一下为什么我对它如此着迷，以及它和 OpenClaw 之间的关系。

Pi 是由 Mario Zechner 编写的。与 Peter 不同，Peter 的目标是“带一点疯狂的科幻感”，而 Mario 的风格则非常务实。尽管他们的风格差异很大，但 **OpenClaw 和 Pi 实际上遵循的是同一个理念：**

> LLM 非常擅长写代码和运行代码，因此应该拥抱这一点。

从某种程度上来说，这也不是偶然，因为 Peter 去年把我和 Mario 一起拉进了这个“agent 兔子洞”。

# 什么是 Pi？

Pi 是一个 **编码 agent**。

而现在已经有很多这样的工具。事实上，我认为现在随便挑一个现成的工具，你都可以体验到所谓的 **agentic programming（Agent 式编程）**。

在这个博客的其他文章里，我曾经积极评价过 **AMP**。我之所以喜欢 AMP，是因为它给我的感觉是：这是一个由真正沉迷于 agent 编程的人打造的产品，而且他们尝试过很多不同方案，最后才形成现在的设计，而不是只是给某个技术套上一个华丽的 UI。

而 Pi 对我来说之所以特别，有两个主要原因：

* **第一，它的核心非常小。**
  它拥有我所见过的 agent 中最短的 system prompt，并且只有四个工具：
  `Read`、`Write`、`Edit`、`Bash`。

* **第二，它通过扩展系统弥补了这个极小的核心。**
  这个扩展系统允许扩展在 session 中持久化状态，而这是一件非常强大的事情。

还有一个额外的小优点：**Pi 本身写得像一款真正的软件。**

它不会闪烁界面，不会占用大量内存，不会莫名其妙崩溃，非常稳定。它显然是由一个对软件质量非常在意的人写出来的。

Pi 同时也是一组小组件的集合，你可以在其之上构建自己的 agent。OpenClaw 就是这样构建的，我自己的 Telegram bot 也是这样做的，Mario 用它构建了 [mom](https://github.com/badlogic/pi-mono/tree/main/packages/mom)。

如果你想构建一个连接到某个系统的 agent，只要把 Pi 指向它自己以及“mom”，它就会帮你生成一个。

# Pi 中没有什么

要理解 Pi 中**有什么**，其实更重要的是理解 **Pi 中没有什么**，以及为什么没有，更重要的是：**为什么未来也不会有。**

最明显的缺失是对 MCP 的支持。

Pi **完全不支持 MCP**。

如果你愿意，你当然可以写一个扩展来支持它。或者你也可以像 OpenClaw 那样，通过使用 **mcporter** 来支持 MCP。mcporter 会把 MCP 调用暴露为 **CLI 接口或者 TypeScript 绑定**，你的 agent 或许可以用它做点什么。也可能不会，我也不知道 :)

但这并不是因为懒得实现。

这是 **Pi 的设计哲学**。

Pi 的整个理念是：如果 agent 还不会做某件事，你不应该去下载一个扩展或者技能。**你应该让 agent 自己扩展自己。**

Pi 推崇的正是：**写代码并运行代码**。

当然，这并不意味着你不能下载扩展。系统完全支持这样做。但 Pi 更鼓励的方式是：你可以让 agent 参考某个已有扩展，例如：“像那边那个扩展那样实现，但做一些我喜欢的修改”。

# 为 Agent 构建的 Agent

如果你观察 Pi（以及 OpenClaw）在做的事情，你会发现一种像“粘土”一样可塑的软件形态。

这种理念对系统架构提出了一些要求，而这些要求必须在设计之初就被考虑进去。

例如：

Pi 的底层 AI SDK 被设计为：一个 session 可以包含来自多个不同模型提供商的消息。系统意识到 session 在不同模型提供商之间的可移植性其实是有限的，因此它不会过度依赖任何某个模型提供商的特定功能。

其次，除了模型消息之外，session 文件还会保存 **自定义消息**。这些消息可以被扩展用于保存状态，或者由系统本身用于维护一些信息——这些信息可能完全不会发送给 AI，或者只发送其中的一部分。

因为有了这样的机制，并且扩展状态可以持久化到磁盘，Pi 还具备 **热重载（hot reloading）** 能力。也就是说，agent 可以写代码、重新加载、测试，然后不断循环，直到扩展真正工作。

系统还附带文档和示例，agent 自己也可以用这些内容来扩展自己。

更有意思的是：**Pi 的 session 是一棵树结构。**

你可以在 session 中创建分支并在不同分支之间导航。这带来了很多有趣的可能性。例如：当某个 agent 工具坏掉时，我可以在一个新的分支里修复它，而不会污染主 session 的上下文。工具修复之后，我可以回到原来的位置，而 Pi 会总结在另一条分支里发生了什么。

这很重要，因为例如在 **Model Context Protocol** 中，MCP 工具通常需要在 session 开始时被加载到系统上下文中。这会导致一个问题：如果你重新加载工具，就可能破坏缓存，或者让 AI 对之前的调用行为产生混乱。

# 上下文之外的工具

在 Pi 中，一个扩展可以注册一个工具，让 LLM 可以调用它。有时我确实会这样做。

例如，虽然我对 **Beads** 的实现方式有一些批评，但我确实认为，让 agent 能访问一个待办事项列表是很有用的。因此我让 agent 给自己实现了一个本地的 issue tracker。

因为我希望 agent 也能管理待办事项，所以在这个场景中我选择给它提供一个工具，而不是 CLI。目前这是我唯一加载到上下文里的额外工具。

但大多数时候，我给 agent 添加的东西要么是 **skills**，要么是 **TUI 扩展**，用来让我和 agent 的交互更愉快。

Pi 的扩展不仅可以提供 `/` 命令，还可以直接在终端中渲染自定义的 TUI 组件，例如：

* 旋转加载动画
* 进度条
* 交互式文件选择器
* 数据表格
* 预览面板

这个 TUI 系统非常灵活，以至于 Mario 证明了你甚至可以在里面运行 Doom。

## `/review`

随着越来越多代码由 agent 编写，在 agent 自己没有进行审查之前就把未完成的工作扔给人类，其实并没有太大意义。

因为 Pi 的 session 是树结构，我可以在一个新的分支里进行代码审查，然后把修复带回主 session。

这个界面设计参考了 Codex，可以方便地查看：

* commit
* diff
* 未提交修改
* 远程 PR

我还让 prompt 特别关注我关心的问题，比如：提醒我新增的依赖。

---

## `/control`

这是一个我正在实验但没有日常使用的扩展。它允许一个 Pi agent 向另一个 Pi agent 发送 prompt。它本质上是一个非常简单的 **multi-agent 系统**。

---

## `/files`

这个扩展会列出 session 中修改或引用过的所有文件。你可以：

* 在 Finder 中打开
* 在 VS Code 中查看 diff
* 快速预览
* 在 prompt 中引用

例如 `shift+ctrl+r` 可以快速预览最近提到的文件。

# 软件正在构建软件

上面这些扩展只是一些例子。

关键是：**这些都不是我写的。**

它们是 agent 按照我的要求写出来的。

我只是告诉 Pi：“帮我写一个扩展。”
然后它就写出来了。

这里没有 MCP，没有社区技能库，也没有任何下载来的东西。

当然，我确实使用了很多 **skills**，但它们都是我的 agent 自己写出来的。例如，我完全替换了所有浏览器自动化 CLI 或 MCP 工具，改成了一个直接使用 CDP 的 skill。

不是因为那些工具不好，而是因为这样更简单、更自然。**agent 维护自己的功能。**

我的 agent 现在有很多技能，而且重要的是：如果某个技能不再需要，我就会把它删掉。

例如：

* 一个用于读取其他工程师分享的 Pi session 的 skill
* 一个帮助生成 commit message 的 skill
* 一个帮助更新 changelog 的 skill

我甚至还有一个 skill 用来让 Pi 使用 `uv` 而不是 `pip`，同时我还写了一个扩展来拦截 `pip` 和 `python` 调用并重定向到 `uv`。

# 软件构建软件的未来

使用像 Pi 这样的极简 agent，会让你逐渐适应一种新的思维方式：

**使用软件来构建软件。**

如果把这种理念推向极致，你甚至可以移除 UI，把它直接连接到聊天系统。

这正是 **OpenClaw** 在做的事情。

而考虑到 OpenClaw 最近的爆炸式增长，我越来越觉得：**这很可能会成为未来的一种软件形态。**
