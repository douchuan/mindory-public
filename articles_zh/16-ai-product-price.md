# AI 产品成长期定价模型设计
——以 Mindory 为例的商业化增长策略

## 一、背景
Mindory 是一款以**阅读**为核心的 AI 产品。现阶段，我们的第一优先级是**健康增长**，而非短期利润最大化。

但增长，绝不等于盲目烧钱。

一套可持续的成长期定价体系，必须同时满足：
- 降低用户入门门槛
- 保证毛利率非负
- 控制基础设施风险
- 实现可预测的规模化
- 提升长期估值潜力
- 保持未来被收购的吸引力

本文将系统梳理一套**专为 AI 原生产品成长期设计**的定价策略。

---

## 二、成长期核心原则
早期 AI / SaaS 产品的定价逻辑，与成熟 SaaS 完全不同。

### 成长期优先级排序
1. 用户规模
2. 用户留存
3. 使用深度
4. 数据资产积累
5. 商业化变现
6. 盈利

利润不是第一目标，但**健康的盈利模型是硬性要求**。

整体战略方向：
> 低 ARPU × 大规模 × 正毛利率

我们的目标不是最大化单用户利润，而是**搭建一套可规模化的经济系统**。

---

## 三、理解真实成本结构
如果只用下面的公式判断可持续性，一定会被误导：
```
预算 ÷ 单次调用成本 = 最大可调用次数
```

AI 产品的成本，远不止模型推理：
- 模型推理成本
- 服务器与算力成本
- 存储成本
- 日志与分析系统
- 监控系统
- 支付渠道手续费
- 客服与支持成本
- 恶意使用损耗

如果只计算 Token 成本，会严重高估盈利能力。
一个健康的定价模型，必须覆盖**全运营成本**，而不只是 API 费用。

---

## 四、核心风险：重度用户
AI 产品的使用分布极度不均衡：
- 70%–90% 轻度用户
- 5%–20% 中度用户
- 1%–5% 重度用户

如果低价提供无限制使用，重度用户会直接击穿毛利率。

因此，定价必须内置结构性防护：
- 单请求 Token 上限
- 每日额度
- 每月额度
- 合理使用政策
- 请求频率限制
- 滥用检测机制

我们的目的不是限制使用，而是**可预测的成本控制**。
没有防护机制，再低成本的模型也会被 "薅秃"。

---

## 五、推荐的成长期定价结构
推荐采用**三层漏斗模型**。

### 第一层：免费版（获客层）
**目标：最大化新手引导与用户习惯养成**

参考结构：
- 每日 10–20 次交互
  或
- 每月 100 次交互

特点：
- 帮助用户形成产品习惯
- 鼓励体验与尝试
- 降低心理决策门槛
- 接受微利甚至小幅亏损

这一层本质是**增长营销基建**。

### 第二层：基础订阅（核心增长层）
参考定价：
```
9 元/月
```

包含：
- 每月 800–1200 次交互
  或
- 固定 Token 额度
- 合理使用政策

有效原因：
- 绝大多数用户不会触达上限
- 收入部分来自未使用的容量
- 成本包可预测
- 付费转化潜力高

这是**主要收入引擎**。

### 第三层：专业版（重度用户层）
参考定价：
```
29 元/月
```

包含：
- 更高额度或软无限使用
- 更大上下文窗口
- 优先处理
- 更快响应速度

定位：
- 安全承接重度用户
- 提升整体 ARPU
- 保护低阶套餐毛利率
- 防止系统滥用

---

## 六、备选方案：基于 Token 的定价模型
不按交互次数，直接按 Token 额度计费。

示例：
- 免费版：20,000 Token/月
- 基础版（9 元）：200,000 Token/月
- 专业版（29 元）：1,000,000 Token/月

优势：
- 与基础设施成本直接对齐
- 不绑定特定模型
- 成本控制透明
- 模型升级时更容易扩容

Token 模式更精准、更灵活。

---

## 七、收益模型
可持续增长的铁律：
```
LTV > CAC
```

更落地的表达：
```
ARPU × 留存月数 > 单用户总成本
```

举例：
- 单用户月均成本：1.8 元
- 订阅价格：9 元
- 平均留存：3 个月

计算：
```
LTV = 9 × 3 = 27 元
总成本 = 1.8 × 3 = 5.4 元
毛利润 = 21.6 元
```

这是**健康的收益模型**。

成长期定价必须从**全生命周期**评估，而不是按单次调用算账。

---

## 八、面向收购目标的定价策略
如果长期目标包含战略收购，买方重点看的是：
- DAU / MAU
- 增长率
- 留存曲线
- 付费转化率
- 使用深度
- 活跃频率
- 数据基建成熟度

他们不会只盯着短期利润。

最优策略是：
> 适度利润 + 高速增长
而不是：
> 小规模下的极致利润率

---

## 九、风险控制机制
AI 系统必须内置成本防护：
- 单请求 Token 上限
- 用户每日额度
- 每月额度
- 请求限流
- 提示词注入防护
- 上下文截断逻辑
- 异常行为检测

无限制的 AI 系统，最终一定会因滥用而崩溃。
可扩展性不只是基建问题，更是**策略设计问题**。

---

## 十、演进路径（0 → 10万 → 100万用户）
### 早期阶段
- 聚焦新手引导
- 接受较薄的毛利
- 验证留存
- 打磨产品市场契合度

### 增长阶段
- 优化定价档位
- 推出高阶功能
- 提升 ARPU
- 降低流失

### 规模化阶段
- 精细化用户分层
- 推出企业版
- 提升终身价值
- 拓展变现渠道

定价必须**随用户规模同步进化**。

---

## 十一、一个常见误区：工程师思维的成本反推定价
在设计定价模型初期，我也踩过一个非常典型的坑：
> 先算出单次请求的模型成本，再从基建成本反推出价格。

逻辑类似：
```
单次交互成本 ≈ 0.0057 元
5 元预算 ÷ 0.0057 ≈ 877 次调用
```

看上去理性、数学正确。
但这是**典型工程师思维**，不是**产品思维**。

### 为什么成本反推定价是陷阱
它默认：
- 定价应该从基础设施费用推导
- 使用限额应该从预算承受力算出
- 盈利 = 价格 − 成本

这在制造业可行。
在成长期 SaaS 或 AI 产品中**完全不适用**。

用户买的不是：
- Token
- API 调用
- GPU 时长

用户买的是：
- 实际结果
- 时间节省
- 认知提升
- 身份增强
- 情绪价值

定价必须反映**用户感知价值**，而非原始成本。

### 真正的定价问题
正确的思考方式是：
- 产品对用户究竟值多少钱？
- 什么价格能最小化转化阻力？
- 什么结构能最大化付费转化率？
- 怎样保证 LTV > CAC？
- 怎样支撑长期规模化？

成本决定**价格底线**，
价值决定**价格上限**。
成长期定价，就在这两者之间。

### 战略升级
从：
> “5 块钱能调用多少次？”

升级为：
> “什么定价模型能最大化可持续用户增长？”

这一跳，让定价从：
- 工程计算
升级为
- 商业系统设计

从：
- 成本会计
升级为
- 经济模型设计

---

## 十二、结论
Mindory 成长期定价的核心目标：
- 保持极低的入门门槛
- 确保健康的单位经济模型
- 控制重度用户风险
- 支持可预测的基建扩容
- 提升长期估值
- 保持收购吸引力

这不是在算“5 块钱能调用几次”。
而是在设计：
> 一套面向 AI 产品的**可规模化增长经济系统**

增长定价不是一个数字。
它是**增长基建**。
